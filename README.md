# RAG Document Chat

Локальное приложение для загрузки документов, построения поискового индекса и вопросно-ответной работы по содержимому документов. Проект ориентирован на сценарий, где пользователь загружает PDF, DOCX, Markdown или TXT, а затем задает вопросы по выбранному документу или по группе документов.

Основная идея системы: не передавать весь документ целиком в языковую модель, а сначала найти наиболее релевантные фрагменты через поисковый индекс, затем передать эти фрагменты в LLM и получить ответ с опорой на источники.

## Что Делает Система

Система позволяет:

- загружать документы в форматах PDF, DOCX, Markdown и TXT;
- извлекать текстовое представление документа;
- разбивать документ на разделы и фрагменты;
- строить векторный индекс по фрагментам;
- хранить документы, метаданные и индексы локально;
- задавать вопросы по выбранному документу;
- задавать вопросы по папке документов;
- получать ответы в режимах "Кратко", "Подробно", "Строго с цитатами";
- видеть источники и найденные фрагменты, на которых основан ответ;
- сравнивать две версии документа в формате чат-ответа.

## Текущий Пайплайн Загрузки Документа

Когда пользователь загружает документ, система проходит несколько этапов.

### 1. Прием Файла

Пользователь выбирает файл через Streamlit-интерфейс. Система проверяет расширение файла и сохраняет исходный файл во внутреннюю папку проекта.

Поддерживаемые форматы:

- PDF;
- DOCX;
- Markdown;
- TXT.

Ограничение: качество дальнейшего поиска напрямую зависит от того, насколько хорошо из файла удалось извлечь текст.

### 2. Извлечение Текста

Для разных форматов используются разные обработчики:

- PDF разбирается в Markdown-представление и, при возможности, в layout-структуру;
- DOCX преобразуется в Markdown;
- Markdown и TXT используются как уже текстовые форматы.

Технологии:

- PyMuPDF для PDF;
- python-docx для DOCX;
- локальные файлы Markdown/JSON для сохранения промежуточного результата.

На этом этапе система получает текстовую версию документа, с которой дальше работает RAG-пайплайн.

Ограничение: если PDF плохо распознан, содержит сканы, сложные таблицы, много колонок или нестандартную верстку, часть текста может быть извлечена неполно или в неправильном порядке.

### 3. Сохранение Исходника и Метаданных

После извлечения текста система сохраняет:

- исходный файл;
- путь к текстовой версии;
- путь к layout-данным, если они есть;
- название документа;
- версию;
- папку документа;
- тип источника;
- служебный идентификатор документа.

Метаданные документов хранятся локально в SQLite.

Технологии:

- SQLite;
- локальная файловая структура `data/`;
- JSON-бандлы документов.

### 4. Разбиение на Разделы

Текст документа делится на логические разделы. Если в документе есть заголовки, система использует их как ориентиры. Если структура документа выражена слабо, документ может быть обработан как один большой раздел или как набор менее точных блоков.

Зачем это нужно:

- улучшить поиск по смысловым частям документа;
- строить summary по разделам;
- потом поднимать в контекст не только отдельный фрагмент, но и соседние связанные фрагменты.

Ограничение: система не всегда идеально понимает юридическую, конкурсную или табличную структуру документа. Она работает с текстовой структурой после парсинга, а не с полноценной семантической моделью документа.

### 5. Построение Summary Разделов

Для каждого раздела строится краткое summary. Это нужно, чтобы система могла сначала понять, какие разделы документа потенциально относятся к вопросу пользователя.

Технологии:

- Yandex Foundation Models / YandexGPT;
- HTTP API;
- локальное сохранение summary.

Этот этап использует внешний LLM API, поэтому скорость зависит не только от компьютера, но и от:

- скорости сети;
- времени ответа Yandex API;
- лимитов сервиса;
- количества разделов в документе.

Ограничение: summary является сжатым представлением раздела. Оно помогает искать релевантные места, но не заменяет полный текст документа.

### 6. Нарезка Документа на Чанки

После разбиения на разделы документ нарезается на чанки - небольшие фрагменты текста. В текущей конфигурации используется размер чанка около 1800 символов с перекрытием около 200 символов.

Зачем нужны чанки:

- языковой модели нельзя эффективно передавать весь большой документ целиком;
- поиск работает точнее на небольших фрагментах;
- ответ можно строить по нескольким найденным кускам текста.

Технологии:

- LangChain text splitters;
- локальные структуры документов LangChain.

Ограничение: если ответ требует собрать информацию из десятков разных мест документа, обычный top-k RAG может найти не все нужные чанки.

### 7. Построение Embedding

Для каждого summary и каждого чанка система считает embedding - числовой вектор, который описывает смысл текста.

Технологии:

- HuggingFace embeddings;
- модель `ai-forever/sbert_large_nlu_ru`;
- Sentence Transformers;
- локальные модели из папки `models/`;
- режим `HF_LOCAL_FILES_ONLY=1`.

Сейчас embedding считается локально на CPU. Это значит, что этап "Добавляю чанки в векторную базу" использует вычислительные мощности компьютера, на котором запущено приложение.

Ограничение: на обычном компьютере этот этап может быть медленным, особенно при большом документе. На сервере с сильным CPU или GPU он может работать заметно быстрее, но для GPU нужна отдельная настройка.

### 8. Запись в Векторную Базу

После вычисления embedding фрагменты записываются в локальную векторную базу.

Технологии:

- ChromaDB;
- SQLite внутри Chroma;
- локальная папка `my_chroma_db`.

В базе хранятся:

- embedding чанков;
- embedding summary;
- служебные метаданные;
- связи с исходным документом.

Ограничение: Chroma используется как локальная база. Это удобно для демонстрации и локальной разработки, но для промышленного многопользовательского режима может понадобиться отдельная серверная архитектура.

### 9. Сохранение Карточки Документа

После индексации система сохраняет полную карточку документа:

- исходные метаданные;
- raw text;
- разделы;
- summary;
- чанки;
- пути к исходным и обработанным файлам.

Эта карточка нужна, чтобы при следующем запуске не пересоздавать весь документ с нуля.

## Пайплайн Ответа на Вопрос

Когда пользователь задает вопрос, система работает не как обычный чат с полным документом, а как RAG.

### 1. Выбор Области Поиска

Пользователь может спрашивать:

- по выбранному документу;
- по папке документов.

Если выбран один документ, поиск идет только по нему. Если выбрана папка, система собирает индексы нескольких документов и ищет по ним.

### 2. Анализ Запроса

Система определяет тип вопроса и готовит поисковые варианты запроса. Для повышения качества могут использоваться:

- нормализация запроса;
- исправление опечаток;
- генерация дополнительных формулировок;
- разложение сложного вопроса на подзапросы;
- поиск по summary разделов.

Технологии:

- YandexGPT для некоторых этапов улучшения запроса;
- BM25 для лексического поиска;
- ChromaDB для векторного поиска;
- Sentence Transformers для embedding.

### 3. Поиск Релевантных Фрагментов

Система ищет фрагменты двумя основными способами:

- векторный поиск по смысловой близости;
- BM25-поиск по словам и терминам.

Затем результаты объединяются и ранжируются.

Технологии:

- ChromaDB similarity search;
- BM25Retriever;
- Reciprocal Rank Fusion;
- reranker `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1`.

Ограничение: поиск возвращает ограниченное число наиболее релевантных фрагментов, а не весь документ. Это хорошо для точечных вопросов, но может быть недостаточно для запросов вида "перечисли все..." или "найди все случаи...".

### 4. Расширение Контекста

После выбора лучших чанков система может добавить соседние чанки. Это нужно, чтобы ответ не строился на вырванной из контекста фразе.

Пример: если найден фрагмент с пунктом документа, соседний фрагмент может содержать начало или продолжение этого пункта.

### 5. Генерация Ответа

Собранный контекст передается в LLM. Модель отвечает только по предоставленному контексту.

Технологии:

- Yandex Foundation Models / YandexGPT;
- режимы ответа:
  - "Кратко";
  - "Подробно";
  - "Строго с цитатами".

В ответе система показывает источники и, при включенной настройке, найденные фрагменты.

Ограничение: если нужная информация не попала в найденный контекст, LLM ее не увидит и не сможет корректно включить в ответ.

## На Что Система Заточена

Текущая архитектура лучше всего подходит для точечных вопросно-ответных сценариев.

Хорошие типы запросов:

- "Какая дата подачи заявки?";
- "Какие требования к публикациям руководителя?";
- "Какие документы нужно приложить?";
- "Что сказано про отчетность?";
- "Какие ограничения указаны в разделе X?";
- "Что изменилось между двумя версиями?";
- "Найди пункт про финансирование";
- "Какие условия участия указаны в документе?".

В этих случаях система ищет релевантные фрагменты и обычно дает хороший ответ, потому что вопрос имеет локальный ответ в нескольких местах документа.

## Где Есть Ограничения

Система хуже подходит для задач, где нужно гарантированно пройти весь документ и собрать все элементы одного типа.

Проблемные типы запросов:

- "Перечисли все важные даты в документе";
- "Найди все сроки";
- "Собери все требования";
- "Перечисли все основания для отказа";
- "Составь полный реестр всех документов";
- "Вытащи все суммы";
- "Найди все упоминания организации";
- "Сделай исчерпывающую таблицу по всему документу".

Почему это ограничение возникает:

- RAG выбирает top-k релевантных фрагментов, а не весь документ;
- даты, сроки или требования могут быть разбросаны по разным разделам;
- embedding-поиск может найти наиболее похожие фрагменты, но пропустить другие;
- LLM отвечает только по контексту, который ей передали;
- слово "важные" является оценочным, поэтому модель может дополнительно отфильтровать часть найденной информации.

Для таких задач лучше использовать отдельный extraction-пайплайн: проход по всему raw text, регулярные выражения или NER, сбор кандидатов, дедупликация, затем LLM для оформления результата. В проекте уже есть часть идей под такие сценарии, но основной чат сейчас ориентирован именно на RAG-поиск.

## Пример с Датами

Если спросить:

```text
Какая дата подачи заявки?
```

Система, скорее всего, найдет релевантный фрагмент и ответит хорошо.

Если спросить:

```text
Перечисли все важные даты в документе
```

обычный RAG-пайплайн может вернуть только часть дат, потому что он не сканирует весь документ целиком. Он выбирает несколько наиболее релевантных фрагментов и строит ответ по ним.

Это не означает, что дат в документе нет. Это означает, что текущая архитектура лучше работает с точечными вопросами, чем с полным извлечением всех сущностей по документу.

## Пайплайн Сравнения Документов

В системе есть режим сравнения двух версий документа.

Общая логика:

1. Пользователь выбирает левую и правую версию документа.
2. Система загружает сохраненные карточки документов.
3. Разделы двух документов сопоставляются между собой.
4. Определяются измененные, добавленные и удаленные части.
5. Пользователь задает вопрос в формате чата.
6. LLM формирует ответ по результатам сравнения.

Технологии:

- локальные JSON-бандлы документов;
- SQLite-реестр документов;
- алгоритмическое сравнение секций;
- YandexGPT для формулирования ответа.

Ограничение: качество сравнения зависит от качества разбиения документов на разделы. Если парсинг PDF нарушил структуру, сравнение тоже может быть менее точным.

## Производительность

Время обработки документа зависит от нескольких факторов:

- размер документа;
- количество страниц;
- качество PDF;
- количество разделов;
- количество чанков;
- скорость CPU;
- наличие или отсутствие GPU;
- скорость ответа Yandex API;
- скорость диска.

Самые тяжелые этапы:

- парсинг сложных PDF;
- построение summary через Yandex API;
- вычисление embedding для всех чанков;
- запись embedding в ChromaDB;
- reranking при поиске.

Сейчас embedding и reranker работают локально на CPU. На сильном сервере эти этапы могут быть быстрее. На сервере с GPU ускорение может быть значительным, но для этого нужно отдельно настроить использование CUDA.

## Используемые Технологии

Интерфейс:

- Streamlit.

Парсинг документов:

- PyMuPDF;
- python-docx.

RAG и обработка текста:

- LangChain;
- LangChain text splitters;
- LangChain Chroma integration;
- LangChain HuggingFace integration;
- BM25Retriever;
- Sentence Transformers.

Векторный поиск:

- ChromaDB.

Модели:

- `ai-forever/sbert_large_nlu_ru` для embedding;
- `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1` для reranking;
- YandexGPT через Yandex Foundation Models API для summary, улучшения запросов и генерации ответов.

Хранение:

- SQLite;
- JSON;
- локальная файловая структура проекта.

## Локальный Запуск

Установить зависимости:

```powershell
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

Запустить интерфейс:

```powershell
.\.venv\Scripts\python.exe -m streamlit run app.py --server.address 127.0.0.1 --server.port 8501
```

Открыть:

```text
http://127.0.0.1:8501
```

## Настройки Окружения

Для работы нужны переменные окружения в `.env`:

```env
YANDEX_CLOUD_FOLDER=your_folder_id
YANDEX_API_KEY=your_api_key
EMBEDDING_MODEL_PATH=path_to_local_embedding_model
RERANKER_MODEL_PATH=path_to_local_reranker_model
HF_HOME=path_to_local_huggingface_cache
HF_LOCAL_FILES_ONLY=1
```

Важно: API-ключ Yandex должен иметь доступ к Yandex Foundation Models.

## Ключевая Идея Для Презентации

Проект демонстрирует локальную RAG-систему для работы с документами:

- документы загружаются и превращаются в текст;
- текст разбивается на управляемые фрагменты;
- фрагменты индексируются в локальной векторной базе;
- пользователь задает вопрос;
- система находит релевантные фрагменты;
- LLM формирует ответ только по найденному контексту;
- пользователь видит источники ответа.

Главное ограничение: это не система полного юридического аудита документа и не универсальный extractor всех сущностей. Текущая версия лучше всего подходит для точечных вопросов, поиска по смыслу, навигации по документу и демонстрации RAG-подхода.

## Возможные Направления Развития

Для дальнейшего улучшения можно добавить:

- отдельный режим полного извлечения дат, сроков, сумм и требований;
- батчевую индексацию с более подробным прогрессом;
- поддержку GPU для embedding и reranker;
- более точную обработку таблиц;
- отдельный pipeline для OCR-сканов;
- расширенную диагностику качества парсинга;
- экспорт ответов и найденных фрагментов в отчет;
- серверную версию Chroma или другую production-ready векторную базу;
- систему ролей и многопользовательский режим.
